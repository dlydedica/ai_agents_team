## Принципы взаимодействия

### 1. Самоорганизация
Команда не имеет жёсткой иерархии. Orchestrator определяет, какие агенты нужны для конкретной задачи. Если задача простая — Orchestrator может пропустить Architect и сразу отдать задачу Developer.

### 2. Асинхронность
Агенты работают независимо. Каждый агент получает подзадачу и возвращает результат. Orchestrator собирает результаты воедино.

### 3. Вложенность
Любой агент может сам выступать в роли мини-Orchestrator'а для своей подзадачи, привлекая других агентов.

### 4. Итеративность
Если Reviewer находит проблемы — задача возвращается Developer на доработку. Если Architect видит архитектурные риски — возвращается на перепроектирование.

## Формат задачи

Задача передаётся в структурированном виде:

```json
{
  "id": "task-001",
  "title": "Добавить REST API для пользователей",
  "description": "Описание задачи...",
  "context": {
    "project_type": "fastapi",
    "stack": ["python", "fastapi", "postgres"],
    "existing_code": "путь к проекту"
  },
  "constraints": ["срок", "стиль кода", "зависимости"]
}
```

## Формат результата

```json
{
  "task_id": "task-001",
  "status": "completed",
  "artifacts": {
    "code": ["src/api/users.py", "src/models/user.py"],
    "tests": ["tests/test_users.py"],
    "docs": ["docs/api/users.md"]
  },
  "summary": "Добавлен REST API для пользователей: CRUD, JWT-аутентификация"
}
```
