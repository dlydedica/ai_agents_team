## Уровни иерархии

### Уровень 1: 🧠 CEO (Главный Оркестратор)
- Видит задачу целиком
- Принимает стратегические решения
- Не погружается в детали реализации каждого отдела

### Уровень 2: 👔 Heads (Руководители отделов)
- Получают подзадачу от CEO
- Управляют своим отделом
- Отвечают за результат своего направления
- Могут взаимодействовать с другими Heads напрямую

### Уровень 3: 👥 Сотрудники
- Выполняют конкретную работу
- Отчитываются своему Head
- Могут взаимодействовать с сотрудниками других отделов по согласованию

## Принципы взаимодействия

### 1. Иерархия коммуникации
```
CEO → Head → Employee
```
Head может делегировать Employee право прямого общения с другим отделом.

### 2. Самоорганизация отделов
Каждый Head сам решает, как распределить работу внутри отдела. CEO не диктует, кто именно будет делать задачу.

### 3. Совет отделов
При сложных cross-department задачах CEO созывает совет Heads для координации.

### 4. Итеративность
Если QA находит баги — задача возвращается в Development. Если Architecture видит риски — возвращается на перепроектирование.

## Типовые сценарии

### Сценарий 1: Новый микросервис
```
CEO → Product (анализ) → Architecture (проект) → 
     Development (код) → QA (тесты) → 
     DevOps (деплой) → Docs (документация) → CEO
```

### Сценарий 2: Хотфикс бага
```
CEO → Development (фикс) → QA (проверка) → 
     DevOps (деплой) → CEO
```

### Сценарий 3: Редизайн UI
```
CEO → Product (требования) → Design (макеты) → 
     Development (вёрстка) → QA (проверка) → CEO
```
