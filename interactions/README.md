# 🔄 Модель взаимодействия отделов

## Общая схема

```
  ┌─────────┐
  │  Задача  │
  └────┬─────┘
       │
  ┌────▼─────┐
  │    CEO   │  ←── Анализ, планирование, назначение Heads
  └────┬─────┘
       │
       ▼
  ┌──────────┐     Handoff     ┌──────────┐     Handoff     ┌──────────┐
  │  Отдел A │ ──────────────► │  Отдел B │ ──────────────► │  Отдел C │
  │          │   артефакты     │          │   артефакты     │          │
  └──────────┘                 └──────────┘                 └──────────┘
       │                            │                            │
       │◄──── feedback / issues ────┤◄──── feedback / issues ────┤
       │                            │                            │
       └────────────────────────────┼────────────────────────────┘
                                    │
                               ┌────▼─────┐
                               │    CEO   │  ←── Сборка результата
                               └──────────┘
```

## Типы взаимодействия

### 1. Последовательный Handoff (Sequential)
Отдел завершает работу → передаёт артефакт следующему отделу.

```
Product → Architecture → Development → QA → DevOps → Docs → CEO
```

### 2. Параллельное взаимодействие (Parallel)
Несколько отделов работают одновременно над одной задачей.

```
Development (код) ──┐
Design (UI) ────────┼──► QA (интеграция)
Data (аналитика) ───┘
```

### 3. Совет отделов (Council)
Heads собираются для решения cross-department проблем или архитектурных решений.

### 4. Ревью / Итерация
QA → возврат в Development. Security → возврат в Development.
Architecture → возврат в Product.

## Формат Handoff (что передаётся между отделами)

Каждый handoff — структурированный пакет:

```json
{
  "handoff": {
    "from": "architecture",
    "to": "development",
    "artifacts": ["docs/architecture.md", "specs/api.yaml"],
    "context": "Рекомендации по реализации...",
    "constraints": ["Срок: 2 дня", "Стек: FastAPI"],
    "decisions": ["ADR-001: Выбрана PostgreSQL"],
    "open_questions": ["Оптимизация индексов — решить на этапе реализации"]
  }
}
```

## Принципы

1. **Асинхронность** — отделы не ждут друг друга, если нет зависимости
2. **Прозрачность** — CEO видит статус каждого отдела
3. **Feedback loop** — любой отдел может вернуть задачу на доработку
4. **Escalation** — неразрешимый спор → CEO / Совет отделов
5. **Single Source of Truth** — артефакты хранятся в проекте, не в переписке
