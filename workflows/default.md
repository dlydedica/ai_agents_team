# Default Workflow (Стандартный процесс)

Используется для большинства задач — от новых проектов до фич и багфиксов.

## Flow

```
    Orchestrator                     Orchestrator
    (анализ)                            |
        |                               |
        ▼                               ▼
    Architect ──────► Developer ──────► Reviewer
        |               |                |
        |               |           [нужны правки?]──да──► Developer
        |               |                |
        |               |                ▼
        |               |           Documenter
        |               |                |
        └───────────────┼────────────────┘
                        ▼
                   Orchestrator
                   (сборка отчёта)
```

## Шаги

### Шаг 1: Анализ задачи (Orchestrator)
- Определить тип задачи (новый проект / фича / багфикс / рефакторинг)
- Определить требуемый стек
- Определить состав команды
- Оценить сложность

### Шаг 2: Проектирование (Architect)
- Архитектура решения
- Структура файлов
- API-контракты
- Выбор библиотек

### Шаг 3: Разработка (Developer)
- Реализация согласно спецификации
- Unit-тесты
- Самопроверка

### Шаг 4: Ревью (Reviewer)
- Code review
- Запуск тестов
- Проверка безопасности и производительности

**Цикл:** Если есть замечания → Developer исправляет → Reviewer проверяет снова

### Шаг 5: Документация (Documenter)
- README (если новый проект)
- API-документация
- Комментарии

### Шаг 6: Сборка результата (Orchestrator)
- Собрать все артефакты
- Сформировать итоговый отчёт
- Передать результат
