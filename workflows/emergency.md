# Emergency Workflow (Срочный процесс)

Используется для горящих задач: критических багов, срочных хотфиксов.

## Отличия от Default

| Параметр | Default | Emergency |
|----------|---------|-----------|
| Product | Полный анализ | Пропускается |
| Architecture | Полный проект | Пропускается (упрощённый план) |
| Development | Полная реализация | Минимальное изменение |
| QA | Полное тестирование | Только smoke-тесты |
| DevOps | Полный деплой | Быстрый хотфикс |
| Design | Полный дизайн | Пропускается |
| Docs | Полная документация | Только changelog |

## Flow

```
🧠 CEO (быстрый анализ)
   │
   ├──► 💻 Development (минимальный фикс)
   ├──► 🧪 QA (quick check)
   ├──► ⚙️ DevOps (быстрый деплой)
   └──► 📖 Docs (только changelog)
         │
         ▼
   🧠 CEO (результат)
```

## Шаги

### Шаг 1: Быстрый анализ (CEO)
- Определить критичность (P0/P1)
- Оценить impact
- Определить минимальный состав отделов

### Шаг 2: Хотфикс (Development)
- Tech Lead ставит задачу разработчику
- Минимальное изменение без рефакторинга
- Без новой функциональности

### Шаг 3: Quick Review (QA)
- Проверка только изменённых строк
- Smoke-тесты

### Шаг 4: Деплой (DevOps)
- Хотфикс в прод / staging

### Шаг 5: Результат (CEO)
- Отметить в changelog
- Сформировать отчёт
