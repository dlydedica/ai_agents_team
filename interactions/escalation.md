# 🚨 Эскалация и разрешение конфликтов

## Когда эскалировать

| Ситуация | Эскалация к |
|----------|-------------|
| Требования неясны / противоречивы | Product → CEO |
| Архитектурный спор между Heads | Совет отделов |
| Сроки под угрозой | Head → CEO |
| Безопасность под угрозой | Security → CEO (немедленно) |
| Найден критический баг (P0) | QA → CEO (немедленно) |
| Нарушение лицензий | Legal → CEO |
| Межотдельский конфликт | Heads → Совет отделов |
| Решение требует бюджета | Head → CEO |

## Уровни эскалации

```
L1: Head → Head (прямое общение между отделами)
     Пример: Developer → QA уточнить баг

L2: Head → CEO
     Пример: Architecture → CEO о рискованном решении

L3: CEO → Совет отделов
     Пример: спор между Product и Development по срокам

L4: CEO → Внешний контекст
     Пример: задача выходит за рамки компетенций команды
```

## Процесс эскалации

```
1. Head фиксирует проблему (текст + контекст)
2. Пытается решить на L1 (прямое общение)
3. Если не решено → L2 → CEO
4. CEO решает или созывает Совет отделов (L3)
5. Решение фиксируется в docs/decisions/
```

## Правила

- **P0 (критично)** — эскалация немедленно, минуя уровни
- **P1 (важно)** — эскалация в течение часа
- **P2 (нормально)** — эскалация в ежедневном статусе
- **Любой сотрудник** может эскалировать напрямую CEO, если Head не реагирует
